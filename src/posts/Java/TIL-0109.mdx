# TIL [데이터 통신, 서버 및 배포, MVC Architecture]

## 데이터 통신
* 프론트엔드에서 요청을 보내고, 백엔드가 응답을 처리하는 HTTP 프로토콜을 통해 이루어진다.

### 상태 코드
* 200번대 : 성공
    * 요청과 응답이 정상적으로 처리됨.
* 400번대 : 클라이언트 오류
    * 잘못된 요청, 프론트엔드에서 요청 문제 발생.
* 500번대 : 서버 오류
    * 요청 처리 중, 백엔드에서 에러 발생.

### HTTP 메서드
* Create -> POST
* Read -> Get
* Update -> PUT
* Delete -> DELETE

## 서비스 배포와 서버
### 호스팅 서버
* 데이터를 저장하거나, 서비스를 제공하기 위한 서버 공간이 필요
* 호스팅 서버는 공용 컴퓨터 자원을 임대하여 사용하는 방식.

### DNS (Domain Name System)
* 숫자로된 IP 주소를 사람이 읽기 쉬운 이름으로 변환
    * 192.168.9.1 -> www.example.com

### CORS (Cross Origin Resource Sharing)
* 클라이언트와 서버가 다른 도메인에서 실행될 때 발생하는 보안 이슈
* Spring Framework에서 이를 해결하기 위한 설정 제공.

### SSH (Secure Shell)
* 네트워크 상에서 데이터를 암호화하여 안전하게 원격 접속을 제공하는 보안 프로토콜을

## 전체 흐름도
* 프론트엔드는 사용자가 상호작용할 UI를 담당.
* 백엔드는 데이터 처리 및 서버 로직을 처리
* HTTP 프로토콜을 통해 프론트와 백엔드가 통신.
* 서비스는 호스팅 서버를 통해 배포되며, DNS로 접속을 간소화.
* 배포된 서버는 AWS와 같은 클라우드 환경에서 운영.

## MVC Architecture
* 백엔드 영역에서 주로 사용되는 소프트웨어 디자인 패턴, 애플리케이션을 세 가지 역할로 분리하여 유지보수성, 역할 분리, 확장성을 높이는 구조.

### 주요 구성 요소
1. Model
    * 데이터와 비즈니스 로직
    * 데이터베이스와의 연결, 처리된 데이터 반환 등 수행.
    * Java Beans : 데이터를 캡슐화하여 구조화된 데이터 처리 및 반환.

2. View
    * 사용자에게 보여지는 화면단.
    * HTML과 CSS를 기반으로 데이터를 시각적으로 표현
    * JSP : Java + HTML, 동적 웹 페이지 생성.

3. Controller
    * 사용자 요청을 처리하고, 적절한 모델과 뷰를 연결
    * 비즈니스 로직과 뷰 사이의 중개 역할.
    * Servlet: 논리 코드 구현을 통해, 요청 처리 및 데이터 전달.
    * WAS가 Servlet을 실행시키는 환경을 제공.

## WAS (Web Application Server)
* 웹 어플리케이션 서버는 정적 데이터와 동적 데이터를 모두 처리하는 소프트웨어 환경.
* 웹 서버와 달리, Java와 같은 서버 측 언어를 통해, 동적 데이터 생성 가능.

### 구조
* Client - Back-end - Database
* 요청과 응답이 클라이언트, 서버, 데이터베이스를 통해 순환.

### 동작 흐름
* 사용자가 Controller(Servlet)에 요청을 보냄.
* Controller가 Model(Java Beans)을 통해 데이터 처리 진행.
* 처리된 데이터를 View(JSP)에 전달, 사용자에게 결과 출력.
